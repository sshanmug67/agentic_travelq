
You can have multiple workflow files in the same repo, each deploying to a different target:

AWSTemplateFormatVersion: "2010-09-09".github/
└── workflows/
    ├── deploy-aws.yml       ← push to main → deploy to AWS EC2
    ├── deploy-azure.yml     ← push to main → deploy to Azure App Service
    └── deploy-fargate.yml   ← push to main → deploy to AWS Fargate (Phase 2)


Each file controls its own trigger independently:
# deploy-aws.yml
on:
  push:
    branches: [main]        # triggers on main

# deploy-azure.yml
on:
  push:
    branches: [main]        # also triggers on main (both run simultaneously)
  
# OR you could separate by branch:
# deploy-azure.yml
on:
  push:
    branches: [deploy/azure]  # only triggers on a different branch
```

You have full flexibility. Some common strategies:
```
Strategy 1: Same branch, multiple workflows (both deploy on every push)
─────────────────────────────────────────────────────────────────────
  git push main → deploy-aws.yml runs    → AWS EC2
                → deploy-azure.yml runs  → Azure

Strategy 2: Different branches per environment
─────────────────────────────────────────────────────────────────────
  git push main        → deploy-aws.yml      → AWS Production
  git push staging     → deploy-staging.yml   → AWS Staging
  git push azure/main  → deploy-azure.yml     → Azure Production

Strategy 3: Manual trigger for secondary deployments
─────────────────────────────────────────────────────────────────────
  git push main        → deploy-aws.yml (automatic)
  Manual button click  → deploy-azure.yml (workflow_dispatch only)

# Create a IAM role in AWS to grant privilege to EC2 to pull images from ECR.
# Create a file to create IAM role for EC2 to pull images from ECR. THis creates a file in wherever the command is run
@'
{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"Service":"ec2.amazonaws.com"},"Action":"sts:AssumeRole"}]}
'@ | Out-File -FilePath trust-policy.json -Encoding ASCII

# Create a AssumeRole IAM role.
aws iam create-role --role-name travelq-ec2-role --assume-role-policy-document file://trust-policy.json

# Step 2 — Attach ECR read permission to the IAM role created.
aws iam attach-role-policy --role-name travelq-ec2-role --policy-arn arn:aws:iam::aws:policy/AmazonEC2ContainerRegistryReadOnly

# Step 3 — Create instance profile and add role:
aws iam create-instance-profile --instance-profile-name travelq-ec2-profile

#TO check the instance profile, that is created.
aws iam list-instance-profiles

# Attche Role to Instance Profile created.
aws iam add-role-to-instance-profile --instance-profile-name travelq-ec2-profile --role-name travelq-ec2-role

# Attach EC2 Profile created to EC2:
aws ec2 associate-iam-instance-profile --instance-id i-0cf321bb38216cb5e --iam-instance-profile Name=travelq-ec2-profile --region us-east-1

# Check whether the profile is attached to the EC2 instance.
aws ec2 describe-iam-instance-profile-associations --region us-east-1

# to pull the image uploaded to ECR into EC2.
